# 클러스터 대시보드

클러스터 노드 상태 및 Pod 현황을 실시간으로 모니터링하는 방법을 설명합니다.

---

## 개요

클러스터 대시보드는 Kubernetes 클러스터에 등록된 노드의 상태와 Pod 현황을 실시간으로 확인하는 화면입니다. 노드를 카드 또는 리스트 형태로 보여주며, 노드를 클릭하면 상세 정보와 진단 결과를 확인할 수 있습니다.

---

## 화면 구성

### 상단 정보 바

| 항목 | 설명 |
|------|------|
| 마지막 업데이트 | 데이터가 마지막으로 갱신된 시각 |
| 자동 새로고침 | 활성화된 경우 주기적으로 데이터를 자동 갱신 |
| 새로고침 | 클릭 시 데이터를 즉시 갱신 |
| 그리드 / 리스트 | 노드 표시 방식 전환 버튼 (그리드 ↔ 리스트) |

---

## 요약 카드

화면 상단에 클러스터 전체 현황이 카드 형태로 표시됩니다.

| 카드 | 설명 |
|------|------|
| **Ready 노드** | 정상 동작 중인 노드 수 |
| **Drained 노드** | 스케줄러에서 제외된 노드 수 |
| **전체 Pod** | 클러스터의 전체 Pod 수 |
| **실행 중 Pod** | 정상 실행 중인 Pod 수 |
| **Pending Pod** | 대기 중인 Pod 수 |

---

## 노드 목록

요약 카드 하단에 **Ready 노드**와 **Drained 노드** 섹션이 구분되어 표시됩니다.

### 뷰 모드

우측 상단의 **그리드 / 리스트** 버튼으로 표시 방식을 전환할 수 있습니다.

| 모드 | 설명 |
|------|------|
| **그리드** | 노드를 카드 형태로 표시. 가속기 시각화를 포함한 상세 정보 확인에 적합 |
| **리스트** | 노드를 행 형태로 표시. 많은 노드를 한눈에 파악하는 데 적합 |

### 필터

노드 목록은 두 가지 필터로 조회 범위를 좁힐 수 있습니다.

**가속기 시각화 모드**

| 옵션 | 설명 |
|------|------|
| 사용유무 | 가속기 사용 여부 기준으로 표시 |
| Pod별 구분 | 실행 중인 Pod별로 가속기 사용 현황 구분 |
| 그룹별 구분 | 그룹별 가속기 사용 현황 구분 (그룹 범례 표시) |

**디바이스 필터**

| 옵션 | 설명 |
|------|------|
| 전체 디바이스 | 모든 노드 표시 |
| NVIDIA GPU 등 | 특정 디바이스 종류가 있는 노드만 표시 |

---

## 노드 상세 정보

노드 카드(그리드) 또는 노드 행(리스트)을 클릭하면 상세 정보 모달이 열립니다.

### 노드 스펙

| 항목 | 설명 |
|------|------|
| 상태 배지 | Ready / Not Ready |
| 역할 배지 | Control Plane / Data Plane |
| CPU | 탑재된 CPU 코어 수 |
| 메모리 | 탑재된 메모리 (GB) |
| 가속기 | 종류 및 수량 (Accelerator 노드인 경우) |
| 스토리지 | 탑재된 스토리지 (GB) |

### 실행 중인 Pods

해당 노드에서 현재 실행 중인 Pod 목록을 확인할 수 있습니다.

### 노드 진단

Kubernetes 노드 상태 조건을 실시간으로 확인합니다. 각 항목이 **정상**이면 녹색으로 표시되고, 문제가 있으면 경고로 표시됩니다.

| 진단 항목 | 의미 |
|-----------|------|
| **NetworkUnavailable** | 네트워크 불가, CNI 플러그인 문제 등 |
| **MemoryPressure** | 메모리 부족, OOM 발생 가능성 |
| **DiskPressure** | 디스크 부족, 사용량이 임계값 초과 |
| **PIDPressure** | 프로세스 수 제한 도달, 더 이상 PID를 만들 수 없음 |

!!!warning 진단 항목에 이상이 있는 경우
해당 노드에서 새로운 Pod 스케줄링이 차단되거나 기존 Pod가 불안정해질 수 있습니다. 즉시 원인을 파악하고 조치하세요.
!!!

### 스케줄러 제외

노드 상세 모달 하단의 **스케줄러 제외** 버튼을 클릭하면 해당 노드를 Drained 상태로 전환합니다.

- Drained 상태의 노드에는 새로운 Pod가 배치되지 않습니다.
- 기존에 실행 중인 Pod는 계속 동작합니다.
- 하드웨어 유지보수나 드라이버 업데이트가 필요할 때 활용합니다.
